Section 01

Phased Build Plan

The POC is built in four phases across roughly 5–7 working days. Each phase produces independently testable deliverables. Phases are ordered so that earlier outputs are prerequisites for later ones — the pipeline cannot be skipped.

Day 1–2
PHASE 1 — DATA FOUNDATION

Database Schema + Seed Dataset

Establish the dual-layer data store (Neo4j + PostgreSQL/PgVector) and load synthetic SKU lifecycle data. This is the foundation — nothing works without it.

  • Stand up PostgreSQL with pgvector extension; create entities table with self_emb vector(768) and reflect_emb vector(768); add ivfflat indexes.
  • Stand up Neo4j; define all 8 node labels and 8 typed relationship types with property maps.
  • Write Python data loader: reads JSON seed records → inserts Neo4j nodes + Postgres rows atomically.
  • Produce seed dataset: ≥200 entities across all 8 node types, ≥500 typed relationships, covering all required failure patterns.
  • Verify data integrity: every Neo4j entity_id maps to exactly one Postgres row; no orphan vectors.
Exit criterion: SELECT count(*) FROM entities returns ≥200; Neo4j node count matches. All 8 relationship types present in graph.
Day 2–3
PHASE 2 — EMBEDDING PIPELINE

Self-Embeddings + Reflection Vectors + Anomaly Scoring

Build and validate the core innovation: the reflection vector pipeline. The anomaly score query must be running before any agent work begins.

  • Implement embed_entity(entity_dict) using all-MiniLM-L6-v2; populate self_emb for all entities. Verify cosine similarity clusters by entity type.
  • Implement compute_reflection(entity_id): fetch neighbours from Neo4j, apply configurable REL_WEIGHTS dict, weighted-mean aggregate, L2-normalise → write to reflect_emb.
  • Write batch refresh script — full recompute across all entities in a single run.
  • Run anomaly score SQL: 1 - (self_emb <=> reflect_emb) across all SKU entities — verify known problem entities appear in top decile, healthy entities in lower half.
  • Log per-entity anomaly scores to anomaly_score_log table for drift tracking.
Exit criterion: Brand-mismatch cascade SKUs score ≥0.80 anomaly. Confirmed healthy SKUs score ≤0.30. Score distribution is not uniformly high (system not flagging everything).
Day 3–5
PHASE 3 — MULTI-AGENT PIPELINE

Supervisor → Planner → Doer → Critic

Build each agent independently in order. Each has a single responsibility. Wire them together only after each passes its own unit test.

  • Supervisor: LLM-backed NL parser producing a typed QuerySpec dataclass (entity_types, time_window, task_type, traversal_depth). Include retry logic for re-routing on Critic rejection.
  • Planner: Receives QuerySpec, emits an ordered TaskList of Cypher templates and ANN targets per hop. Decision logic for which hops use self_emb vs reflect_emb vs both.
  • Doer: Executes (1) Cypher traversal on Neo4j, (2) self_emb ANN on Postgres, (3) reflect_emb ANN on Postgres. Unions and deduplicates by entity_id. Returns candidate causal chains as ordered entity dicts with timestamps.
  • Critic: Validates each chain: monotonic timestamps, ≥5 entities per hop, confidence = temporal × evidence density × avg anomaly. Rejects below 0.65. Returns top-3 ranked chains.
Exit criterion: Brand mismatch query produces a validated chain with confidence ≥0.81. Critic rejects ≥1 candidate chain per test run. End-to-end latency <60 seconds.
Day 5–7
PHASE 4 — DEMO NOTEBOOK + BENCHMARKS

Jupyter Notebook, A/B Cell, Accuracy + Throughput

  • Write end-to-end Jupyter notebook: seed load → embedding pipeline → 3+ scenario queries → per-agent output → final ranked chains. Must run top-to-bottom in <5 min on a laptop.
  • Add A/B comparison cell: same question → closed-world Cypher (0 results) vs RKG agent (validated chain 0.81). Make the juxtaposition unmissable — side-by-side output with clear visual separation.
  • Report benchmarks: end-to-end latency, anomaly score distribution histogram, Critic acceptance rate, classification accuracy vs ground truth labels (target ≥85%).
  • Cover all 6 scenarios in the notebook (at minimum Scenarios 1 and 4 as mandatory demos).
  • Dry-run the full demo — one person drives, another narrates agent reasoning for one validated chain.
Exit criterion: Notebook runs clean, end-to-end, in <5 min. A/B cell is the visual centrepiece. Classification accuracy ≥85% against ground truth. Live demo rehearsed ≥2 times.
Section 02

System Architecture

The POC uses a three-layer architecture: a dual data store at the base, a vector pipeline in the middle, and a multi-agent query layer on top. Each layer is independently testable and has clear interfaces.

↓ NATURAL LANGUAGE INPUT  ·  ANALYST / DATA SCIENTIST
🧠
Agent 1 — Supervisor
LLM NL parser. Extracts entity types, time window, task type. Emits QuerySpec. Reroutes on Critic failure.
🗺️
Agent 2 — Planner
Builds Cypher traversal plan + ANN targets. Decides per-hop: self_emb vs reflect_emb. Emits TaskList.
⚡
Agent 3 — Doer
Executes Cypher + ANN (self) + ANN (reflect). Unions result sets. Deduplicates. Assembles candidate chains.
✅
Agent 4 — Critic
Validates chains: timestamps, evidence density, confidence score. Rejects <0.65. Returns top-3 chains.
↑ CAUSAL CHAINS + CONFIDENCE SCORES  ·  ↓ QUERYS PECS + TASK LISTS
🔤
Self-Embedding Generator
embed_entity(dict) → all-MiniLM-L6-v2 → 768-d vector → self_emb in Postgres.
🔄
Reflection Pipeline
compute_reflection(id) → fetch neighbours from Neo4j → apply REL_WEIGHTS → weighted mean → L2-norm → reflect_emb.
📊
Anomaly Score Query
SQL: 1 - (self_emb <=> reflect_emb). No rule. Ranks all SKUs. Writes to anomaly_score_log.
↑ ENTITY DATA + NEIGHBOUR GRAPHS  ·  ↓ EMBEDDINGS WRITTEN BACK
🕸️
Neo4j — Entity Graph
8 node labels. 8 typed relationships. Weighted edges. Stores the structural graph for Cypher traversal. Each node has a matching UUID in Postgres.
🗄️
PostgreSQL + pgvector
entities table: self_emb vector(768) + reflect_emb vector(768). ivfflat indexes (lists=100). All ANN searches happen here.
↑ PYTHON DATA LOADER  ·  SYNTHETIC SEED JSON (200+ ENTITIES, 500+ RELATIONSHIPS)
Key design decision: Neo4j owns the structural graph (traversal, hop logic, causal chain assembly). Postgres owns all vector math (ANN search, anomaly scoring). The two are always kept in sync via the entity_id UUID foreign key. The embedding pipeline is the bridge — it reads from Neo4j and writes to Postgres.
Section 03

Multi-Agent Pipeline Detail

Each agent has exactly one responsibility. They communicate via typed dataclasses, not raw strings. The Critic can force the Supervisor to re-route with deeper traversal if all chains are rejected.

AGENT 01
Supervisor Query Orchestrator

Uses an LLM with a structured-output prompt to parse the analyst's natural-language question into a typed spec. The only agent that talks to an LLM.

INNatural language question string
OUTQuerySpec {entity_types, time_window, task_type, traversal_depth}
REOn Critic rejection → increment depth, re-route to Planner
AGENT 02
Planner Query Decomposer

Translates QuerySpec into a concrete execution plan: which Cypher hops to run, in what order, and whether each hop needs self_emb, reflect_emb, or both.

INQuerySpec from Supervisor
OUTTaskList [ {cypher_template, ann_target, hop_index} ]
AGENT 03
Doer Query Executor

Pure execution. No decisions. Runs exactly what Planner sent. Returns raw evidence, not judgements. The only agent that touches both databases.

INTaskList from Planner
OUTCandidate chains [ entity_path[], timestamps[], anomaly_scores[] ]
AGENT 04
Critic Evidence Validator

Quality gate. Three checks: temporal ordering, evidence density, confidence threshold. Returns only what it can stand behind.

INRaw candidate chains from Doer
OUTTop-3 chains {confidence, entity_path, per-hop reasoning} OR rejection signal to Supervisor
# Critic confidence formula confidence = ( temporal_validity # 1.0 if timestamps monotonically increase, else 0.0 * evidence_density # min(entity_count_per_hop) / required_minimum (cap at 1.0) * avg_anomaly_score # mean anomaly score of anchor entities in the chain ) # Thresholds REJECT_THRESHOLD = 0.65 TOP_K_CHAINS = 3 # Classification output if confidence >= 0.80: return "Confirmed Anomaly" elif confidence >= 0.65: return "Needs Review" else: return "Rejected → re-route"

View PostgreSQL Schema (core tables)
CREATE EXTENSION IF NOT EXISTS vector; CREATE TABLE entities ( entity_id UUID PRIMARY KEY DEFAULT gen_random_uuid(), entity_type TEXT NOT NULL, -- 'sku' | 'brand' | 'tenant_sku' | 'merge_event' | ... label TEXT, properties JSONB, -- all domain attributes: UPC, weight, dims, status ... self_emb vector(768), -- own attribute embedding reflect_emb vector(768) -- neighbourhood-view embedding ); CREATE INDEX ON entities USING ivfflat (self_emb vector_cosine_ops) WITH (lists = 100); CREATE INDEX ON entities USING ivfflat (reflect_emb vector_cosine_ops) WITH (lists = 100); CREATE TABLE anomaly_score_log ( entity_id UUID REFERENCES entities(entity_id), anomaly_score NUMERIC(6,4), computed_at TIMESTAMPTZ DEFAULT now() ); -- Rank all SKUs by anomaly — no rule required SELECT entity_id, label, ROUND(CAST(1 - (self_emb <=> reflect_emb) AS numeric), 4) AS anomaly_score FROM entities WHERE entity_type = 'sku' ORDER BY anomaly_score DESC LIMIT 20;
View Reflection Vector Pipeline (Python)
import numpy as np from sentence_transformers import SentenceTransformer model = SentenceTransformer('all-MiniLM-L6-v2') # Configurable — domain team can tune without touching algorithm REL_WEIGHTS = { 'MERGED_INTO': 3.0, 'SCANNED_ON': 2.5, 'MAPS_TO': 2.0, 'TRAINED_WITH': 2.0, 'FUZZY_MATCH': 1.8, 'BELONGS_TO_BRAND': 1.5, 'HAS_PACKAGE': 1.5, 'USED_BY': 1.0, } def embed_entity(entity: dict) -> list[float]: text = f"{entity['type']} {entity['label']} " \ f"{' '.join(str(v) for v in entity['properties'].values())}" return model.encode(text).tolist() def compute_reflection(entity_id, neo4j_session, pg_conn) -> list[float]: rows = neo4j_session.run(""" MATCH (e {entity_id: $eid})-[r]-(n) RETURN type(r) AS rel, n.entity_id AS nid """, eid=entity_id) signals = [] for row in rows: n_emb = fetch_self_emb(row['nid'], pg_conn) weight = REL_WEIGHTS.get(row['rel'], 1.0) signals.append(weight * np.array(n_emb)) if not signals: return [0.0] * 768 vec = np.mean(signals, axis=0) return (vec / (np.linalg.norm(vec) + 1e-8)).tolist()

Section 07

Risk Register & Mitigations

RiskSeverityMitigation
Reflection vectors are uniformly high — system flags everything, anomaly score loses signal HIGH Validate score distribution before building agents. Tune REL_WEIGHTS dict. Ensure ≥10 confirmed healthy SKUs have low scores as a calibration check. Gate Phase 3 on this passing.
Neo4j–Postgres sync drift — entity_ids exist in one DB but not the other after partial loads HIGH Write atomic loader: Neo4j insert + Postgres insert in a single transaction wrapper. Add integrity check script that counts both and asserts equality before any pipeline run.
LLM Supervisor produces malformed QuerySpec — agents downstream receive bad types MEDIUM Use structured output mode (JSON schema enforcement) on the LLM call. Add Pydantic validation on QuerySpec. Fallback: if parsing fails, use a hardcoded default QuerySpec for the demo scenario.
Notebook runs too slowly — >5 min on a standard laptop, violates AC 18 MEDIUM Pre-compute and cache all embeddings before notebook starts. Use pickle or NPZ file. Notebook loads pre-computed vectors, does not call the sentence encoder live during the demo run.
Critic accepts all chains — no rejections visible, AC 10 fails MEDIUM Seed at least 3 synthetic "bad chains" with out-of-order timestamps or low entity counts. These should always fail. Demonstrate Critic rejection on these explicitly in the notebook.
Classification accuracy below 85% — AC 14 fails MEDIUM Tune anomaly score thresholds against the 20 ground-truth labels during Phase 2. Adjust the confirmed/needs-review/rejected cutoffs (0.80/0.65) based on observed score distribution on labelled data.
Seed data doesn't produce enough signal variation — failure cases don't separate from healthy in score space LOW Design seed data with intentionally extreme attribute divergence for failure cases. Brand mismatch SKUs should have no training images and multiple scan failures — maximally different neighbourhoods from healthy SKUs.
A/B comparison not visually compelling — judges miss the core demo moment LOW Use notebook markdown + colour-coded output cells. Print closed-world result first ("RESULT: 0 rows"), then RKG result with confidence score prominently formatted. Make it impossible to miss.
Decision point: At the end of Phase 2, before writing a single line of agent code, run the anomaly score query against all seed entities. If known-bad SKUs are NOT in the top decile, stop and fix the seed data or the REL_WEIGHTS before proceeding. The agent pipeline is worthless without a signal to work from.
